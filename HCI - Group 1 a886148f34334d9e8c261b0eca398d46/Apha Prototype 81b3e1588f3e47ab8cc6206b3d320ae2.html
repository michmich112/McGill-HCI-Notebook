Apha Prototype

This deliverable explores our product, VideoWise (previously SeniorityRules), through its phases of design iteration and the alpha system development/deployment.

The Alpha System can be found at this website.

Evolution

In light of Group 21's formative feedback (can be found here) on our project in addition to the feedback in response to our system presentation, a few decisions and changes were made to our system. The following section discusses these changes along with the justification in terms of usability or feasibility.

Design Evolution

The following are design decisions that were implemented to our computer prototype following feedback based on user evaluation tests performed by both Group 21 and additional tests performed by us. For clarity, we restate our usability goals and target heuristics:

Usability Goals:

[U1] Users of the tool should be able to adopt it without any required technical background.

[U2] Emotions should be shown seamlessly by the tool when they are required.

[U3] Emotions should be understandable to the user.

[U4] Users should feel a sense of connection, community, or family by the system.

[U5] The system should not require the use of a dedicated device.

Neilson's Heuristics:

[H1] Error prevention (Nielson's Heuristics)

  • We wish to deter users from causing errors to occur in the system. Given that we worry that our target demographic will not be able to fix problems if they occur in certain cases, we plan to reduce the number of possible things that can go wrong for the user.

[H2] Aesthetic and minimalist design (Nielson's Heuristics)

  • We believe that it is imperative to ensure that the UI provided to our user base is as streamlined as possible to ensure that they don't confuse different actions. As such, we intend to use a particularly minimalist design.

[H3] Help users recognize, diagnose, and recover from errors (Nielson's Heuristics)

  • We believe that it is imperative that users understand when something has gone wrong and we also think that in general, it should be fixed by itself because our target audience may have a more limited ability to troubleshoot.

[H4] Help and documentation (Nielson's Heuristics)

  • We intend to provide clear and well laid out help for each of the actions a user can be taken. We feel it is a real possibility that users will take a few tries to remember how to perform actions so we want to clearly lay them out in our help section which will appear on each page.

[H5] Few features beyond absolute necessities (Nielson's Heuristics)

  • We feel that including too many options to the system will have an inverse effect in that users will not be able to use any of them due to confusion. As such, we intend to only include those which are absolutely necesary.

[H6] Clearly describable (Nielson's Heuristics)

  • We believe that it is important that users be able to explain problems to troubleshooters. As such, icons should be clear and describable - especially when errors arise - to allow troubleshooters to help fix problems.

Based off of these goals and heuristics we wish to achieve, the following changes were made. The changes can be seen in our computer prototype found here.

  1. Making a Call

    The problem: Terminology ambiguity and security risks.

    Usability Goals: U1, U4

    Heuristics: H1, H5, H6

    All usability tests illuminated general confusion regarding making a call by inputting a username or userID and creating an 'empty' call and joining an 'empty' call. A large part of this confusion stemmed from ambiguity surrounding what 'empty' call meant and unclear wording of the task test (the latter of which will be discussed in the Test Procedure Evolution section). The caused extra questions from the users or inability to complete the task. In addition, with our current functionality, our design allows for the option for users to brute force calling people by manually entering in possible usernames and disrupting the lives of our users. Due to the reality that our target users (elderly aged people) are frequently the subject of scams and other harmful malpractices, this especially poses a key security risk.

    Design Change 1: We removed the option for users to call someone directly via inputting a new username.

    Design Change 2: We rebranded 'empty calls' to 'video call rooms'. We felt this concept more closely resembled Zoom's terminology, which many of our target users have some experience with thus making the concept easier to understand. We haven't gotten a chance to verify this hypothesis of reduced confusion from different terminology, but will be a key part of our upcoming round of iterative user testing.

    Design Change 3: The 'Make Call' button directs the user to a screen with only two buttons: 'Dial' and 'Help'. In addition to these buttons, directions are provided to the user on what the button, 'Dial', will do (start a video call room) and what will happen after the video call is started (being shown the room ID and directing the user to send it to another person). We decided to keep the button that actually starts a video call as 'Dial' because it was clear after the user testing that the buttons with 'Dial' were always clear to the user in what the expected behavior of clicking that button would be.

    Note: The 'Join Call' button and flow has not been changed. This was because the confusion came solely with 'starting an empty room' and not 'joining' one.

  1. Help Button Tooltip

    The problem: No users throughout any user test made use of the 'help' button.

    Usability Goals: U1, U4, U5

    Heuristics: H1, H4, H6

    Users never used the 'Help' button despite it being clear and present at every stage of the prototype and despite the fact that there were tasks that users were confused about. The users only explored the 'Help' button when it was specifically pointed out to them after all the user test tasks had been completed.

    Design Change 4: We added a 'tool tip' highlight feature when a user first logs into the system. In our computer prototype (using Framer), this was implemented right at the beginning when someone opens up the prototype link to simulate a first time use of the system. See the image below for a visual of the tooltip and the contents that is conveyed to the user.

  1. Inputting in the Call ID

    The problem: When inputting in the room ID for 'Join Call' process-flow, there was a functionality error with entering in the numbers into the fields that resulted from Framer's own limitations.

    Usability Goals: U1, U5

    Heuristics: H1, H2, H5

    As shown in the left picture below in Figure 2, the expected functionality of having 5 separate input bars for entering the 5-digit ID is that after entering each number, the cursor would automatically move to the next bar. Framer does not support this functionality and, as such, every user encountered the same problem of typing too many numbers in the first bar, having to delete some numbers, and then click to type in the next bar.

    Design Change 5: The input text field was changed from 5 separate input bars to one input bar because Framer does not provide the functionality for automatically switching to the next input bar as shown in the picture on the left. We would like to revisit this design, however, because we feel it clearly illustrates the 5-digit nature of the code and where/how to fill out the numbers properly.

  1. Increased Accessibility

    There are many different types of colorblindness. In our initial studies, we found evidence to use colors that have high contrast with each other. Because of this and a desire to keep a familiar sense of what the colors red and green convey, the colors we use primarily in this system are limited to red, green, blue, black, and white (as can be seen in our prototype. However, it was brought to our attention that red-green color blindness is one of the more common forms of color blindness.

    Design Change 5: Already, we prioritized text over icons, but now we revisited all our components of the computer prototype to ensure that regardless of one's ability to see color, it would be clearly communicated to the user. See figure 3 below for an example that was implemented in our Alpha System to better communicate the state of the system. Additionally, we are committed to keeping the font of the text at large as the space and aesthetic design allows us and as easy to read (H2). In the prototype, this meant that all font was at least size 16+, all font was sans serif for increased readability, a dedicated background color, and plenty of line spacing. (Used the following as resources: WCAG 2.0, Designing for Senior Citizens, and

    How to Design for Color Blindness).

    (Future) Design Change 6: The final change, which we will not implement right now due to time constraints, is introducing a 'Color Blind Friendly' button in our system. The button would change our system to colors that are optimized for those with red-green color blindness. (See here for some examples).

Evaluation Heuristics Changes

In response to the suggestions made by Group 21, we update our heuristics as follows. Specifically, we include a new heuristic of "Match between system and real world."

Heuristics

We wish to achieve the following heuristics in our system. Several are taken from Nielson's sources and are attributed as such:

7. Match between system and real world (Nielson's Heuristics)

  • In order for our target group to have a smooth transition into our product, which may be complicated by a lack of technological expertise, it is important that our product be similar to real world products they have used before. Specifically, we wish to include as many allegories to traditional phone calls as possible. Our system is thus intended to mimic as much of the functionality of a normal phone call as is possible.

Reporting and Test Task Changes

Based on the feedback provided to us by Group 21 during their formative feedback section, we update the test tasks in a couple of ways. The most relevant is that we forgot to include specific instructions about how to test the emotion detection system. In this updated version, we include a specific method of testing to see if our emotion system is clear to users and is able to provide an improved experience in cases in which bandwidth for a call is not sufficient to provide consistent video. The updated tasks are as follows:

Below are the tasks a user will be asked to perform during the Usability Test. In the Usability Test Reporting document (provided below), each task group will have the individual steps listed in the document for clarity.

  1. Making a call to a known user
    • In this test have the user start from the Home Page
    • Ask the user to make a new call to their friend who's username is SteveWonder.
    • Once on the call ask them to mute themselves.
    • Then ask them to Hang Up
    • Make sure to take out your stopwatch to capture the time it takes the user to complete the task.
  1. Start an empty call
    • In this test have the user start from the Home Page (shown Previously)
    • Ask the user to make a new call without a specific partner to call.
    • This, ideally, will direct them to the "Start an empty call" button within the "Join Call" menu.
    • Once on the call ask them to report the call id that they should send to their friend.
    • Then ask them to turn their video off.
    • Finally ask them to Hang Up.
    • Make sure to take out your stopwatch to capture the time it takes the user to complete the task.
  1. Joining an empty call using the call ID
    • In this test have the user start from the Home Page (shown Previously)
    • Ask the user to join the call created by their friend SteveWonder.
    • SteveWonder's user ID is 12345.
    • The ID of that call is 67890.
    • Make sure to take out your stopwatch to capture the time it takes the user to complete the task.
  1. Using the emotion detection system
    • This system will be on by default at this stage in the development.
    • Check to see the user's response. Ensure that they are able to understand the purpose of the system.
    • If necessary, ask the user what the reason for the system is to gauge their level of understanding of the feature.
  1. Understanding the emotion system *
    • After reading the 'Info message' presented to the user, ask the user what the purpose of the emotions is.
    • As the evaluator, when an emotion appears, say quick blurb, story, or sentence that would communicate the displayed emotion (i.e. 'I'm so happy that we finally have a time to talk!' for 'Happy', or 'I'm having a really hard day at work today.' for 'Sad') and and ask the user whether they feel the shown emotion represents what the user is saying.
    • Attempt to ascertain - with follow up questions if necessary - whether the user knows who the emotion belongs to as well as what their partner is feeling based on the emotion.
    • Log each of the phrases used as well as the emotion displayed as a result and the response from the user viewing the emotions.

    * Note that the emotion in the computer prototype needs to be manually triggered by the evaluator by clicking the status bar while in a 'video call room' started through any method (i.e. make a call, join a call, recently called). Then once the 'Happy' emotion is displayed, the evaluator can cycle through 4 different emotions by continuing to press the status bar.

In addition, we removed the "heuristic survey' from our test plan as it was not beneficial in the way we thought it might be. We now ask that testers keep our heuristics in mind when performing the normal testing rather than using the survey to account for how successful we were in attaining our goals with the heuristics.Screenshot from 2020-11-16 18-42-32

Our key objectives for this project are as follows:

  1. Make a system that will preserve emotional information about a video call when the video is not available (poor connectivity, no camera, etc.) through sentiment analysis
  1. Make a system whose design is easily understandable and easy to operate by elderly individuals.

Our Usability Goals are reiterated, again:

[U1] Users of the tool should be able to adopt it without any required technical background.

[U2] Emotions should be shown seamlessly by the tool when they are required.

[U3] Emotions should be understandable to the user.

[U4] Users should feel a sense of connection, community, or family by the system.

[U5] The system should not require the use of a dedicated device.

A reminder that our target persona, at this stage, is the 'Eager Beaver' who is ready to take on the new challenges of learning how to use new technologies, but can get easily overwhelmed. However, once our Eager Beaver can use it correctly once, they are confident in their skills moving forward.

  1. The alpha system implements the same feature designs (minimal buttons and choices, the same color scheme, concise yet explicitly stated text, etc.) to ensure the system is easy to use and understand.

    [Main Objective 2, U1]

  1. Our system properly allows for making a video call room and joining a video room as stated earlier in Design Changes 1, 2, and 3.

    [U1, U5]

    [U1]

While baseline functionality of our alpha system prototype has been achieved, we indicate below some limitations facing our web application. Each point stated below will be completed for the Beta System.

  1. When the sentiment is categorized, currently we only display the emotion name (i.e. 'Happy', 'Sad', and 'Neutral') based on the sentiment analysis processing. While having the image of the emotion in addition to the name of the emotion is key for conveying the lost emotional information of a video call, we are working to first achieve the proper sentiment classification during a conversation and to display it visually in accordance with our heuristics (namely Aesthetic and minimalist design; clearly describable).
  1. Currently, we were unable to completely implement video quality monitoring due to unforeseen complications. This will be fixed for the Beta System.
  1. Our alpha system currently does not support creating a user account. Once this is successfully made, our Design Decision 4 will be properly implemented to highlight the 'Help' button for first time users.
  1. The alpha system currently does not support functionality of having a 'Recently Called' section where the user information for the 5 previous users who were called from one user's account is saved for quick dialing. A call can currently only occur through making or joining a video call room.
  1. The alpha system supports functionality for incoming notifications when someone is calling you directly via the 'Recently Called' option.
  1. Once all the above functionality has been successfully implemented, we will do a thorough sweep of the entire system to ensure the design of the text, buttons, colors, and key informational messages adhere to our design decisions outlined earlier in this deliverable as well as in previous deliverables.
