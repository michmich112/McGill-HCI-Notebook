Formative Feedback

Group 1 provided feedback on Group 2's system, Virtual Class Map. This deliverable details our test procedure, analysis of the usability tests, and critiques on the test plan and design choices for the overall system.

Testing

In total, we conducted three usability tests on three different individuals. All participants fall into one of three identified persona groups mentioned in Group 2's latest deliverable. That being, all participants are "students who have been negatively affected by the switch to online lectures using video conferencing tools, or students that can gain from the added interactions provided by the product."

One thing to note, Group 2's 'Usability Test Plan' calls for ideally, 3 examiners present for each usability test. We considered this note, but ultimately decided that 1 examiner would be sufficient for each usability test. Moreover, due to COVID19 restrictions, some of the tests were conducted in person (with roommates) and some were conducted over Zoom. Because the test procedure and reporting sections of Group 2's deliverable did not call for tests to be visually and audibly recorded, only one of the tests was recorded by video - this video is being provided as an example of our test.

  1. Download V-Class

    This was accomplished by following the steps outlined in the 'User Manual'. Please see our 'Design Critiques' section below for more details on this step.

  1. Prepare Data Collection Sheet

    The data collection sheet provided was a picture of the necessary information. On one screen the prototype was displayed and on the other screen the test results were recorded.

  1. Begin Usability Test with 'User Introduction' sheet

    Show 'User Introduction' sheet to user (we used the sheet provided in the previous deliverable stage as no sheet was provided during this stage)

  1. Ask user for their consent

    The consent form is presented to user and completed. Then ask for permission to record (if test was recorded).

  1. Introduce application, its' features, and main problem
  1. Explain goal of Usability Test to user

    This is stated to be "[determining] whether the design meets the goals set during the test creation." Additionally, evaluators explain to user that they will abstain from helping unless directly asked for assistance.

  1. Ask user to complete the 'Pre-test Questionnaire'
  1. Guide user through test tasks while recording information

    For each task, we time how long it took user to complete the action.

    For some tasks, we record how many incorrect clicks were made before the correct action was completed.

    Additionally, observations of things the user said or did were made.

    General observations by the evaluators were also noted down.

  1. User Comments

    Once test tasks completed, user was prompted to make any statements or comments about the system.

  1. Ask user to complete the 'Post-test Questionnaire'
  1. Heuristics Usability Evaluation

    Once the form completed, the examiner asked the user to rate and select the heuristics that they thought were applicable to the prototype. The following heuristics were evaluated:

    • Recognition rather than recall
    • Match between the system and the real world
    • Flexibility and efficiency of use
    • Error prevention

Post-Test Results

The following is the results for post-test questionnaire.

Analysis

First, we wish to discuss the fact that the measurement criteria for the tasks across all three participants were comparable (within a few seconds of each other for all tasks). Our team views this a positive indicator for the usability goals that deal with understandablity (goals 1, 3, 4, and 5) of the system.

Second, while each task was completed with none to minimal trouble, a few issues stood out. There was no 'info' or 'help' button at any stage of the system. This was especially obvious in the tests given that all three users expressed confusion at what the log in criteria was/should be. One of the users immediately started entering their McGill email, as they normally would to use Zoom, and typed random numbers for the Meeting ID. However, if this was not intended or if username is not supposed to be the email, etc. this was not clearly communicated. This is likely a problem that is less important at this stage as any input would successfully complete the login process. However, when this is not going to be the case - support should be provided.

Third, our group feels that the signifiers and mappings of the buttons and seating chart seem to be, for the most part, appropriately designed. This is exemplified by the statement made User 1 made where she said "I like to sit in the front of the class" and then proceeded clicking the remaining available seat 'to sit at'.

Fourth, even though we do feel the signifiers and mappings of the system are appropriate, there was confusion surrounding the difference between the two chats. Despite all users ultimately completing the task with no issues or incorrect actions, there was overall ambivalence that resulted in a longer time for this task than the task for writing in the 'all chat'. We deduce that this was caused by unclear nomenclature to distinguish, without a shadow of a doubt, between the two chats. This uncertainty will result in less people using the neighborhood chat feature if it is left unaddressed. Additionally, there are some general design issues in the system that are discussed in further detail in the section 'Design Critique' below.

Fifth, after reviewing the pre and post test questionnaires, the group feels confident that this system is on the right direction towards solving the issue of student interaction in online classes. The feedback was, on the whole, positive and there were some interesting notes about using this feature especially as an in-between to the 'all chat' feature and standard 'break-out rooms' in Zoom. However, as another participant noted in the post-test questionnaire, the usefulness of this system is strongly related to network effects where the more that people join a 'neighborhood', the more beneficial it is to everyone. If only one student joins a 'neighborhood', then it is the same as no one joining a 'neighborhood'. When considering how to position the product to ensure maximum use for the target users, this should be a key consideration further down the line. There are many considerations and implications that are not clearly stated which will likely have a direct effect on the usage of the system. For example, will teachers be able to read the chat messages? Or is there any benefit to sitting in the 'front' of the room versus the 'back' of the room like in real life (i.e. will a professor be able to see who is sitting where)?

Finally, the following notes are our analysis for the heuristics evaluation of the system:

  • Recognition rather than recall: the use of buttons that are the same as the ones in zoom make identifying actions easier. Furthermore, the placement of those buttons and the extra chat make it easier to recognize where things are.
  • Match between the system and the real world: since we select our seats and get people in a chat based on that, it‚Äôs very similar. The one thing though is that since this is a digital platform we would expect some features such as adding or removing people to the neighborhood chat (just like texting in class).
  • Flexibility and efficiency of use: not being able to change seats or leave the neighborhood area without leaving the meeting. The lack of selection for the neighborhood chat (cannot add or remove people manually). Two different chats.
  • Error prevention: Both chats look the same and there is no quick and easy way to see in which chat you are typing (the caret being too small). There didn‚Äôt seem to be a way to delete a message that had been sent on accident. Could not find a way to get help outside of asking the examiner. There were two buttons to leave, do both of them do the same thing?

Test Plan Critique

In several cases, the test plan was able to accurately assess whether the prototype was able to complete the stated usability goals within the system:

  • The log in and choose a seat features was clearly able to test whether a user was able to choose a seat and select it quickly - the first two usability goals put forth by the team. While the team will likely be able to use the information obtained from this testing phase to determine whether the prototype passed these goals, it may have been beneficial for the team to specifically define an upper bound of time for a user to complete these tasks. This would have made it even more clear whether the prototype was successful.
  • The testing of which chat to be used and whether the user was able to tell the difference will likely lead to fruitful results w/r/t whether users can generally understand how to use both the normal and all chat features. In the case that a user was confused about which was which, it would likely be caught by the evaluation team.
  • The goal of easy navigation between zoom's base functionality and seating functionality was generally well implemented. The seating setup was clearly tested as was the question of whether users would be able to understand whether they were choosing to sit with other people or whether they would be sitting 'alone' based on their preferences.

Some of the functionality of the testing was either difficult to assess or the tests did not look into in enough detail.

  • Due to the nature of the prototype, it was difficult for a user to have a sense of whether they had successfully used the all class chat. Because this portion of the prototype was not fully coded - likely as it will simply be implemented with the stock zoom chat function in the final product - it was hard for participants to understand whether they had successfully completed the 'write to all chat' function. In addition, this made it difficult for testers to know if/when users had been successful. This is not a specific problem with the prototype - rather it is caused by the inconsistency in setup. In addition, this is also a failure of the user manual which included no visuals to teach the testing team about the prototype.
  • It was not possible to test whether usability goal 5 had been completed. Due to the fact that none of zoom's actual functionality was included, a tester would not be able to ascertain whether this goal had been reached. This is likely due to the fact that it is past the scope of this level of testing but this should likely have been specified beforehand.
  • When testing the difference between joining the meeting alone or in a neighborhood, it was somewhat unclear at what point the timer should begin. Given that users had to leave the meeting and then again login, it should have been more obvious at what point the timing should begin so that an accurate relation between the time taken for the two tasks was possible.

With regards to the logistics of testing materials and procedure plan, there were a few items that were not clearly provided. This included:

  • The 'User Introduction' sheet. This was retrieved from the group's previous deliverable.
  • A user consent form. This was also retrieved from the group's previous deliverable.
  • There was no direction to record the test.
  • It was unclear whether the usability heuristics were to be conducted by the evaluators or by the users.

Design Critique

This prototype implements a good design but lacks some essential features for proper usage.

  • Although the user manual specified instructions for installing the prototype on windows computers, it does not function (c.f. screenshot below). Even after following all the troubleshooting instructions provided in the user manual we were unable to make it function properly. We resolved to deploying the project to a private cloud to allow all our evaluators and testers test the application.
  • No instructions specified what features were implemented and which were not. Since no instructions in the testing guidelines or the user manual explained how to use the prototype it was concluded that the all chat was not implemented as there was no way of entering a message in what we recognized to be the "normal" zoom chat (seen below). To fix this, we suggest improving the 'User Manual' with additional sections.
  • The lack of a "help" section in the application is a clear flaw as there is no for the user to find help or clarify points. First use tutorials are also not present, this was seen in the test with users 1 and 2.
  • There is some interference with the multiple similar buttons as well as the two chats. As you can see in the screenshot below there are two "leave" button. Both looking alike. To fix this, we suggest making a decision to have only one "leave" button.
  • The positioning of the "Leave" button right below the "Send" button as seen on the above picture makes it possible for the user to easily miss the "Send" button and accidentally leave the meeting. Students who leave a class meeting for any reason (e.g. slow network, laggy video or audio, accidentally leaving the meeting‚Ä¶) are more likely to not return to that meeting.
  • The presence of both chats at all time is also a design flaw as it doubles the screen real estate that the chat usually occupies when one is using Zoom. One should be able to toggle one, both or none at all. This is especially prominent when a user selects the "Join Class Alone" option and is presented with a "normal" Zoom chat and an empty "neighborhood chat (c.f. screenshot below).
  • Furthermore the ability to leave a group/neighborhood without leaving the meeting is crucial for usability especially in the specified context. As stated above, students who leave a class meeting for any reason (e.g. slow network, laggy video or audio‚Ä¶) are more likely to not return to that meeting.
  • Wrong affordance present given for the user to understand how neighborhood chats are decided. From its name and the fact that a user must choose a seat, it is understood that the users that are included in the neighborhood chat are users that are directly next to the seat we chose. But whether we choose the seat in screenshot 1 or screenshot 2 (c.f. below), the users in the neighborhood chat stay the same.
  • Finally it is impossible to change the users that are in a neighborhood chat. The user may want to add or remove a user to the neighborhood chat but we found ourselves in the inability to do such an operation.
