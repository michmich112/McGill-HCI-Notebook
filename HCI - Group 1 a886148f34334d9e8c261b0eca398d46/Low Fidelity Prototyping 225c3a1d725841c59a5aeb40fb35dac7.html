Iterations

Iteration 1: 10+10 method

We started using the 10+10 method to create multiple sketches of our products. These sketches allowed us to experiment with mobile and desktop layouts. This was extremely important as we want our tool to be as accessible as possible and one of the key points we observed in the previous phase is that our users will tend to be more comfortable with a specific device or type of device. As such, designing for both mobile and desktop environments is a key stake for this project.

These sketches allowed us to identify key decisions we had to make concerning the product:

  • Handling Group calls V.S. 1-1 calling
  • Web app V.S. Native app
  • Login and Sign up processes
  • Physical V.S. Software product
  • Closed Captioning
  • Start and join call without account

Iteration 2:

In this second iteration, we focused on our decisions made at the end of the previous iteration. As such we focused on designing an interface for

  • smooth start and join of call,
  • Handling only 1 on 1 calls,
  • using a web app that is added to a m,obile device's home screen

We also investigated workflows and interactions of the different designs for different processes.

Groupings

Once this second iteration completed we grouped up all the sketches into different categories to identify what we wanted to use for our prototypes.

Usability Goals and Benchmark Tests

Usability Goals

In order to determine which of the prototypes to continue with, we developed a series of goals which we believe will be important to complete in order to create a successful system. They are as follows:

1. Users of the tool should be able to adopt it without any required technical background.

  • Because our target demographic is likely to be less technologically comfortable than the user base of an average video conferencing product, we believe it is vital for the tool to not require much knowledge of technology to use effectively.

2. Emotions should be shown seamlessly by the tool when they are required.

  • In order to allow for the tool to take the place of clear video in times of low bandwidth, we believe it is important that emotions should be displayed without much input from the user when they are required.

3. Emotions should be understandable to the user.

  • In the cases where sentiment analysis is produced, users should be able to have a good idea of what emotion is being displayed by the tool.
  • Though it is not vital that the user gets the exact emotion (i.e. happy vs. excited) it is important that a user be able to verify the general idea easily.

4. Users should feel a sense of connection, community, or family by using the system.

  • When using the emotion system, users should be able to feel more connected with the person on the other side of the call - especially when the video is not able to run effectively.

5. The system should not require the use of a dedicated device

  • The system should be able to run on an everyday device like a laptop - no extra hardware required.

Summary of Test Results

Two users participated in this study. User A falls into the persona group, the Underdog. User B falls into the persona group the Eager Beaver. Both interviews lasted around an hour each with about 10-25 minutes of preparation beforehand. Users were instructed to perform the tasks as outlined in the 'Test Tasks' above and consented to being recorded. The main concerns being tested with our users was (1) ease of use and (2) understanding the meaning of our emotion avatars.

User A had some confusion at the beginning with understanding how the prototypes represented technological devices, but got the hang of it after a while. The 'help' button was utilized frequently by User A and User A was also willing to experiment with the different buttons to find the action they were looking to perform. User A seemed to be less confused by Prototype 2 due to its more 'regular phone calling' capabilities. User A stated very positively at the end that having a tool that will be 'easy to use' would be great to have. User A verbally stated that they understood what the emotion avatars were representing after further explanation, but this insight indicates that it is likely that more work should be done to clearly convey why one's video might not work and what the avatar is doing to replace it.

User B's comfort in operating the prototypes decreased from Prototype 1 to Prototype 2 as more options (video/emotion avatar) and more buttons (on/off camera, audio, etc.) were introduced. User B was confused by the meaning of the buttons (i.e. with the messaging feature) and stated that many buttons were unnecessary, but did state that some features were beneficial if designed more clearly (i.e. messaging in a call).

Analysis

As can be seen within the data collection section above, users were able to complete almost all of the tasks we tested during tests of the first prototype. This likely represents a system which is well suited to users - even those with very limited technical background. The system had far fewer options for the types of actions that could be taken, however, that does not necessarily correspond to a more useful system. As can be seen within the Post-Test Survey Results above, our users indicated that they believed it was more important for the system to be easier to use even at the cost of these options.

With regard to the second prototype, the users had more trouble executing several of the tasks we laid out for them within their tests. Of specific importance to our project, the need to turn on the emotions feature when the audio/video of the call dropped out was one that caused difficulty for our users. This is an important point as we will likely proceed with creating a system which does not require the user to turn on the emotions feature in order for it to begin working.

As can be seen within the Post-Test Survey Results above, the users felt that the emotions system would generally make them feel more connected to the other user. This is of particular importance to our project as it is the main thrust of our work.

As we touched on above, we find that the first prototype - which takes away much of the responsibility of using the system away from the user - conforms much better to our user base. This is likely due to the fact that our users will often have lower technological knowledge than other segments of the population. Providing them with a tool which is more streamlined and easier to use will likely prove more useful to them than providing a large range of options and expecting them to be able to use the options. We will be proceeding with this in mind moving forward with our project.
